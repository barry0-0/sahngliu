# 咖喱粑粑 - 产业园区商流供应链系统 (S2B2C 轻量版) 项目交接文档 (Handoff Document)

本项目是一个轻量化、去资金网关、以“法务电子合同”为约束纽带、以“线下履约对账及抽佣”为核心的**大宗商品供应链商流平台高保真原型系统**。目前整体品牌主色调已更新为雅致的**紫罗兰色系 (`#ae86e7`)**。

---

## 📂 1. 项目目录结构与技术栈

项目采用 **纯前端原生架构 (HTML5 + CSS3 + Vanilla JS)**，零依赖，直接在主流浏览器中打开即可运行。

```text

│   ├── mall.html                      # 【买家端】PC 商城（港口宽幅 Banner、意向求购、竞价大厅、个人中心）
│   ├── h5.html                        # 【买家端】H5 移动端（小程序化首页、购物车分类结算、订单中心、个人中心）
│   ├── merchant.html                  # 【商家端】PC 后台（数据中心、店铺装潢锁定、商品/上架双菜单、出价定标）
│   ├── merchant-h5.html               # 【商家端】H5 移动后台（店铺看板、商品列表、发货操作、定标管理）
│   ├── admin.html                     # 【平台运营端】园区总控后台（单级导航、店铺审核、供求/公告监控、合规下架）
│   └── assets/                        # 静态资源与前端核心代码
│       ├── css/                       # 样式表（global.css, admin.css, mall.css, h5.css）
│       └── js/                        # 业务逻辑
│           ├── components.js          # 全局 UI 核心组件（通用弹窗、分页器、付款凭证展示、状态秒切代理）
│           ├── mock-data.js           # 统一数据总线（核心内存数据库，同步多端业务状态）
│           ├── mall.js                # 买家端 PC 业务逻辑（求购/竞价/购物车/个人中心）
│           ├── h5.js                  # 买家端 H5 业务逻辑
│           ├── merchant.js            # 商家端 PC 业务逻辑
│           ├── merchant-h5.js         # 商家端 H5 业务逻辑
│           └── admin.js               # 运营端后台业务逻辑（审核流、订单透视、报价核校、竞价监督）
```

## 🔄 2. 系统核心数据流动与联动机理

项目所有数据的存储与变更统一流经 `platform/assets/js/mock-data.js` 中的 `MockData` 对象：
1. **多端状态实时联动**：由于单页面共用同一内存数据库（或跨页面通过 localStorage 同步），各角色端状态是互通的。例如：商户在商家后台下架某商品，买家大厅对应卡片立即下架；商户在中选定标后，买家订单库中立即动态生成竞价类型的待签约订单。
2. **法务强约束流转**：系统不接入第三方支付，而是依靠法务合同进行双边约束。履约链条为：`意向/竞价成交 -> 生成电子合同 -> 双方在线电子盖章 -> 线下公对公汇款对账 -> 卖家发货录入物流 -> 买家确认收货结算`。

## 🛠️ 3. 系统核心重构模块规格说明

### 3.1 订单履约与付款凭证查看机制
* **订单类型标签**：所有订单（买家 PC 个人中心、买家 H5、商家后台、运营后台）均前置突出展示 `现货交易` 或 `竞价交易` 标签，剔除了无用的 `求购转单` 等杂乱类型。
* **货款付款凭证展示**：
  - 订单状态为 `已付款` (status >= 1) 时，在订单详情页面（买家 PC 全屏详情、运营端详情模态框）自动渲染“大宗货款支付凭证”专区。
  - **线下汇款凭证**：展示买家上传的打款凭证文件名称，并提供“查看凭证”按钮，点击调起预览弹窗。
  - **在线快捷支付**：展示托管账户支付的流水号（如 `TXN-PAY-ORDxxxxx`），并提供“查看电子回单”按钮。
* **确认收货权限控制**：
  - **买家端**：仅买家（PC 个人中心及 H5）在订单处于“已发货”状态时显示【确认收货】按钮。
  - **运营端**：提供【代确认收货】按钮，拥有监管放款执行权。
  - **商家端**：**隐藏任何确认收货按钮**，仅保留录入发货物流的权限。

### 3.2 商家端：店铺装潢状态机与商品管理
* **店铺状态机控制**：店铺状态限制为 `正常营业`、`待审核`、`闭店中` 三种。
  - **未开店 / 申诉拒审**：商户可以编辑店铺资料并点击【提交审核】。
  - **待审核状态锁死**：当店铺状态为`待审核`时，**禁用 (disabled) 店铺名称、头像、Banner 输入框和上传按钮**，隐藏提交按钮并暴露【撤回审核】按钮。点击撤回后，状态退回`未开店`并解锁编辑。
  - **闭店中（关停/拒审）**：隐藏所有可操作按钮，展示警告信息与下架/拒审原因。
* **商品与上架独立双列表**：
  - 菜单拆分为独立的【商品列表】与【上架列表】。
  - 两表格首列新增展示“商品 ID”（如 `P001`），次列为自增“序号”。
  - 商品状态统一使用：`待审核`、`已上架`、`已下架`。已上架支持下架；已下架支持编辑与重新提交审核。
* **竞价定标与合同自动生成**：
  - 商家可在后台查看名下竞价公告的出价详情，系统自动按出价金额降序排列。
  - 商户选择高价者点击【选为中标】，竞拍项目状态更新为 `已结束`，系统在后台订单库中自动为该中标人生成一条对应的“竞价交易”待签约订单，拉通闭环。

### 3.3 运营端：公告监控去化编辑、只读实况与模块化审核弹窗
* **去化编辑与流程精简**：
  - 运营端竞价公告监控中**剔除了编辑和删除按钮**，运营人员无法篡改竞价数据，只保留“待审核”时的【审核】按钮，以及“已通过”时的【下架】按钮。
  - 状态标签彻底简化，去除了诸如 `等待公布 (已通过)` 这样带有括弧后缀的冗余标签。审核中展示 `待审核`，过审后直接根据节点显示：`看货报名`、`现场看货`、`参加竞价`、`等待公布`、`竞价已结束`。
* **出价实况监督（只读）**：
  - 运营人员点击公告行的【查看详情】，弹起全新的 `#modal-admin-bid-detail` 弹窗，直观呈现所有买家的报价排行榜（价格降序），**但去除了定标及选择中标人的按钮**，保障平台第三方监督的纯洁性。
* **补全四大物理审核弹窗**：
  - 在 `admin.html` 底部补全了先前缺失的四个物理模态框 DOM：**店铺入驻审核** (`#modal-audit-shop`)、**商品上架审核** (`#modal-audit-product`)、**供求信息审核** (`#modal-audit-demand`) 及 **竞价公告审核** (`#modal-audit-bidding-ann`)。
  - 支持审核结果（通过/拒绝）单选，且拒绝原因输入框限制 50 字以内并带有 `0/50` 动态字数统计。

### 3.4 买家端：个人中心购物车嵌入、极简报价与求购表单
* **导航栏精简与购物车移入个人中心**：
  - 删除了 PC 商城顶部 header 的全局搜索框及购物车角标，保持顶部视觉的极简商务感。
  - 购物车移入买家 PC 个人中心左边栏【我的购物车】子视图，并在菜单右侧新增红点角标（`#cart-badge-uc`）同步显示购物车内商品总量。
* **购物车店铺区块分类**：
  - 购物车列表中商品不再采用扁平列表，而是按照“所属店铺（商家）”进行区块板块化隔离。支持店铺维度的全选与结算，且限制不支持跨商家合并结算。结算按钮文案统一更新为“生成订单”。
* **报价与求购发布表单极简**：
  - **报价表单**：商城个人中心求购大厅及 H5 中，买家（商家角色）进行报价时，**删除了“可供货数量”及“报价备注说明”字段**，表单精简为仅包含一个“报价金额”输入项。
  - **求购发布表单**：买家发布需求时，**移除了“备注”输入框**。供求大厅的展示卡片去除了期望价和备注，以交期和品名为核心。
  - **供求报价查看权限**：买家发布的需求在收到报价后，买家前台不显示“查看报价”按钮，**报价明细强制仅在运营后台提供查看**，防止买家私下联系绕过平台。
* **“只看我发布”与“只看我参与”筛选**：
  - 从买家个人中心移除了冗余的求购列表菜单，改为在大厅/寻源页中提供“只看我发布”与“只看我参与”复选框（互斥）。
  - 勾选“只看我参与”时展示已报价的需求卡片，显示“我的报价”并支持对未采纳的报价进行一键加价修改。

## 🚀 5. 二次开发与联调部署说明

1. **零构建部署**：本项目为纯前端高保真系统，可直接通过 `Live Server`（或本地 Nginx/http-server）托管 `platform/` 目录进行演示及交互验收。
2. **全局事件秒切说明**：为配合业务效果演示，在任意页面点击诸如 `待付款`、`待发货`、`正常营业` 等带有状态类名或标签色彩的元素时，页面会自动在对应的预设状态集中循环轮转文字与色彩。二次开发若需要接入真实接口数据，可在 `components.js` 的事件委托中屏蔽这一演示机制。
3. **数据扩充**：若需要拓展或修改初始数据，可直接修改 `platform/assets/js/mock-data.js` 中 `MockData` 对象的初始化数组。
