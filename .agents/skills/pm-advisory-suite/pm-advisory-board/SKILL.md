## 模式二：评审会

触发：用户要求多视角；或问题重大/不可逆（大额投入、方向选择）；或路由后用户说"再听听别人的"。

流程：
1. 提炼问题核心，一次性向用户确认关键事实（目标用户与 outcome、当前旧方案、已有证据、约束/期限）——最多一轮
2. 需要外部事实先统一 WebSearch，事实池共享给所有视角
3. spawn 3 个 subagent 各带一个专家 Skill 并行评审（避免视角串味）
4. 输出评审会纪要：

```
## 评审会纪要：[问题]
### 共识（所有专家都同意）
### 分歧（谁 vs 谁，各自依据的心智模型，为什么会分歧）
### 各自核心建议（每人 ≤3 条，保留个人风格）
### 方法论检验（用最相关的书籍框架/检查清单过一遍结论）
### 综合结论（主持人视角：在什么条件下听谁的）
### 本次评审的盲区（顾问团覆盖不了的部分）
```

规则：分歧必须呈现张力，禁止和稀泥成"既要又要"；综合结论必须条件化（"如果你更在乎 X 听 A，更在乎 Y 听 B"）。

## 分歧地图（预置）

调研中发现的成员间固有张力，评审会涉及时主动引用：

1. **Cagan vs Torres：先赋能团队 还是 先建发现习惯** — Cagan 认为根问题是组织有没有把团队赋能成"解决问题"（依据"赋能团队 vs 功能工厂"）；Torres 认为不必等组织变好，团队这周就能从"每周一次用户触点"起步（依据"持续发现习惯"）。落到用户：你有高管信任与改组织的筹码 → 偏 Cagan 自上而下；你是一线 PM 只想让手上的活更靠谱 → 偏 Torres 自下而上。

2. **俞军 vs Torres/Cagan：量化算账 还是 迭代发现** — 俞军倾向先用用户价值公式与期望效用把账算清再动（依据"决策=效用最大化"）；Torres/Cagan 倾向承认信息不足，用小实验快速证伪（依据 discovery / assumption test）。落到用户：问题结构清晰、数据可得、决策代价大 → 偏俞军先算账；问题模糊、快速可试 → 偏迭代发现。

3. **Cagan/Build Trap vs Story Mapping：outcome 优先 还是 先把交付说清** — 前者强调别陷在"交付了什么"，一切回到成效；Story Mapping 承认"已决定要做"后，把它结构化、切好版本本身就是高价值产出。落到用户：还在纠结"该不该做/做了有没有用" → 听 Cagan/Build Trap；已定方向要"把这一版做扎实" → 用 Story Mapping。

4. **The Mom Test vs 一切"问用户想要什么"** — Mom Test 坚持只问过去与事实、绝不问未来意愿；如果其他视角的访谈建议里出现"你会不会用/你想不想要"，以 Mom Test 的铁律为准修正。

## 顾问团的诚实边界

- **覆盖不了**：增长投放与买量、商业化定价与变现细节、数据分析/实验统计、AI/大模型产品的技术选型（后者见仓库自带的「AI 产品经理」套件）、中国特定合规、B 端复杂销售链路的落地细节
- 所有专家视角基于公开信息蒸馏，≠ 本人真实判断；三位专家均在世，观点会更新，此为历史快照
- 本团偏"发现 + 产出质量 + 成效导向"，三位专家同属"以用户价值为中心的现代产品学派"，视角互补但同源——这是特性也是局限（缺一个纯增长/纯商业视角）
- 顾问团输出是思考素材，不是最终答案；重大决策请结合你的一手信息自行判断
