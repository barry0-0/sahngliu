# Find Content To Visual

此 skill 只做一件事：

**先找出什么值得画，再决定怎么画。**

它不是最终出图器。

它是 `visualization-lab` 里的前置分析层。

当用户给的是一份材料、一个模块、一个项目、一套规则，而不是已经明确的“某张图”，先用此 skill。

## 适用场景

- 用户说“帮我看看这个文档/模块/项目里哪些地方适合可视化”
- 用户说“把这里面复杂内容图解一下”，但还没指定要画哪几张图
- 内容里有大量流程、判断规则、角色协作、状态流转、层级结构、依赖关系
- 直接写成长段文字会难懂，图解能明显降低理解成本
- 用户希望先看可视化方案，再决定是否出图

## 不适用场景

- 用户已经明确指定要画哪张图、画什么内容、要什么格式
- 内容很短，1 段清晰文字或 1 个表格就能说清
- 只是要美化已有图，而不是分析内容

## 核心原则

1. 先理解材料，再判断是否值得可视化。
2. 不为“能画”而画。只有明显降低理解成本时才建议图解。
3. 优先挑复杂度高、阅读阻力大、跨角色/跨阶段/跨规则的内容。
4. 未经用户确认，不直接进入最终出图。
5. 候选点宁少勿滥。默认给 1 到 5 个高价值候选点。
6. 若内容本质上更适合表格、列表、短文，也要明确说“不建议画”。

## 优先识别的可视化类型

看到下面内容时，优先考虑提出图解建议：

- 多步骤且有分支的流程
- 多角色参与且有交接的流程
- 规则判断、准入条件、例外路径
- 模块边界、依赖关系、分层结构
- 状态变化、生命周期、事件驱动流转
- 时序交互、调用链、消息往返
- 多概念之间的映射、包含、对比、归类
- 长段说明文字中反复提到的“先后关系”“因果关系”“责任归属”

## 图型建议映射

发现可视化点后，先给出推荐图型，不直接选实现工具：

- 流程 + 分支判断
  - 优先：流程图
- 跨角色协作 + 责任交接
  - 优先：泳道图
- 多参与方消息往返 + 时间顺序
  - 优先：时序图
- 生命周期 + 状态切换
  - 优先：状态图
- 模块分层 + 依赖边界
  - 优先：架构图 / 分层图 / 嵌套图
- 树状层级 + 父子关系
  - 优先：树图 / 组织图
- 规则集合 + 分类归纳
  - 优先：矩阵 / 象限 / 金字塔 / 树图
- 概念关系网 + 说明结构
  - 优先：概念图 / 架构图 / 嵌套图

最终使用哪个出图 skill，等用户确认后再交给 `../../SKILL.md` 做路由。

## 工作流程

### 1. 明确分析对象

先确定用户要你处理的对象：

- 单个文档
- 某个模块
- 某个项目
- 某段规则说明
- 某份方案材料

同时确定粒度：

- 全量扫描
- 指定章节
- 指定目录
- 指定功能范围

若用户没限定，默认按“先给高价值局部，不做无边界全量梳理”处理。

### 2. 抽取内容结构

从材料中先抽下面几类信息：

- 参与者 / 角色
- 关键对象 / 模块
- 关键动作 / 事件
- 判断条件 / 规则
- 输入 / 输出
- 阶段 / 状态
- 边界 / 依赖 / 例外

不要把原文大段复述一遍。要先把结构抽出来。

### 3. 识别候选可视化点

把材料里“读起来最绕、最密、最容易误解”的部分找出来。

每个候选点至少判断：

- 是否存在明显顺序关系
- 是否存在角色交接
- 是否存在条件分支
- 是否存在层级或包含关系
- 是否存在状态变化
- 是否存在高频术语和概念映射

### 4. 给出候选方案

每个候选点输出以下信息：

- `标题`
- `内容范围`
- `为什么值得可视化`
- `推荐图型`
- `建议优先级`
- `预期解决的问题`

必要时再补：

- `主要元素`
- `是否建议拆成两张图`
- `更适合正式图 / 草图 / 可编辑源文件`

### 5. 等用户确认

分析输出后，停在“候选方案”层。

让用户确认：

- 先画哪一个
- 是否要画多个
- 是否要调整范围
- 是否有指定交付格式或风格

未确认前，不直接调用最终出图 skill。

### 6. 确认后交给 visualization-lab 路由

用户确认后：

1. 把已确认内容整理成规范化出图 brief
2. 读取 `../../SKILL.md`
3. 按主 skill 的路由规则选择最终出图 skill
4. 再进入对应 skill 生成结果

规范化 brief 至少包含：

- 图的目标
- 图的主题
- 内容边界
- 推荐图型
- 核心节点 / 角色 / 关系
- 用户要求的格式、风格、是否可编辑

## 输出要求

默认输出应简洁、可确认，不要直接展开成长报告。

推荐格式：

1. `可视化点 1：...`
   - 范围
   - 推荐图型
   - 为什么值得画
   - 优先级
2. `可视化点 2：...`
3. `可视化点 3：...`
4. `建议先确认：先画哪一个，是否调整范围`

如果没有值得画的点，要明确说明：

- 哪些内容不值得画
- 为什么文字或表格更合适

## 协同规则

- 此 skill 负责“发现与方案”，不负责最终出图落地
- 最终出图一律回到 `visualization-lab` 主 skill 继续路由
- 不要在同一步里既做大范围分析，又直接混用多个出图 skill
- 若用户已经明确“只画某张图”，跳过此 skill

## 不要这样做

- 不要拿到长文档就整篇逐段复述
- 不要罗列 10 个以上候选点造成选择负担
- 不要把简单列表硬说成要画图
- 不要在用户还没确认时直接生成最终图
- 不要把“推荐图型”和“最终实现 skill”混为一谈
